## 1. Dev auth bypasses Supabase RLS everywhere

**Files**: `src/lib/auth.ts`, `src/lib/actions/courses.ts`, `src/lib/actions/materials.ts`, `src/lib/actions/generation.ts`, `src/lib/actions/labs.ts`, `src/lib/actions/instances.ts`, `src/lib/actions/enrollment.ts`, `src/lib/actions/chat.ts`, `src/lib/actions/reviews.ts`, `src/lib/actions/student-lab.ts`

**State**: The dev login page sets a `muto-dev-user` cookie read by `getCurrentUser()`. No real Supabase Auth JWT exists. Every Server Action uses the **admin client** and enforces ownership (`created_by = user.id`, `uploaded_by = user.id`, etc.) in application code.

**Why it's wrong**: RLS is the safety net. With the admin client, a bug in `assertCourseAccess` or a missing ownership filter leaks data across tenants. Fine for a single-dev pilot; unacceptable at scale.

**Fix when real SSO lands**:
1. Replace `getCurrentUser()` with a call to `supabase.auth.getUser()` + a lookup into `public.users`.
2. Switch every Server Action from `createAdminClient()` back to the SSR client (`src/lib/supabase/server.ts`).
3. Audit each action's queries against the RLS policies in `001_initial_schema.sql` and `002_student_rls.sql` — make sure each required access path has a matching policy.
4. Delete `src/lib/actions/dev-auth.ts`, `src/app/login/page.tsx`'s dev UI, and the dev-cookie reads in `src/proxy.ts`.

## 2. `source-materials` Storage bucket is open to the anon role for INSERT

**Files**: `supabase/migrations/005_source_materials_bucket.sql`, `src/components/material-upload.tsx`

**State**: The browser uploads files directly to Supabase Storage using the anon key. Because there is no JWT in dev, the `anon` role is what reaches the server. Migration 005 grants `anon` INSERT on the `source-materials` bucket so the upload doesn't 403. SELECT and DELETE are not granted — reads/deletes go through server code paths that use the service-role key.

**Why it's wrong**: Any caller with the public anon key can write arbitrary files (up to 50MB, limited by MIME allowlist) into this bucket. That's a denial-of-service and storage-cost vector in production.

**Fix when real SSO lands**:
1. Drop the `source_materials_dev_insert` policy.
2. Option A — tighten policy: add an authenticated-only INSERT policy scoped by the first path segment being a courseId the user owns:
   ```sql
   CREATE POLICY source_materials_owner_insert ON storage.objects
     FOR INSERT TO authenticated
     WITH CHECK (
       bucket_id = 'source-materials'
       AND EXISTS (
         SELECT 1 FROM courses
         WHERE id::text = (storage.foldername(name))[1]
           AND created_by = auth.uid()
       )
     );
   ```
3. Option B (stronger) — move the upload through a Server Action that returns a short-lived **signed upload URL** generated by the service-role key. The browser never touches Storage with the anon key. Pattern:
   - `src/lib/actions/materials.ts` gets a new `getUploadUrl(courseId, fileName)` action
   - That action enforces `assertCourseAccess`, then calls `admin.storage.from('source-materials').createSignedUploadUrl(...)`
   - `src/components/material-upload.tsx` calls the action, then PUTs the file to the returned URL
4. Either way, remove the `anon` INSERT policy.

## 3. Multi-file upload is not supported

**Files**: `src/components/material-upload.tsx`

**State**: Single-file upload only. `<input>` has no `multiple` attribute; `handleDrop` and `handleFileSelect` grab `files[0]`.

**Fix**: Add `multiple` to the input, iterate `Array.from(files)` through `handleUpload`. Decide sequential vs parallel based on how the `uploadMaterial` Server Action handles concurrent inserts (should be fine — each creates its own `parse_materials` job).
